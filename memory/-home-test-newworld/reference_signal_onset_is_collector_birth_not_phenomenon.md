**2026-07-20 GFW 探针 ④ 实例（两个会话连续误判、三轮翻案）**：`provider_stop:denied` 事件数「06-13~07-13 整月 0 → 07-14 突增 178 → 07-16 归零 → 07-17 起 355+/天」，被连续两个会话读成「6-7 天前突然爆发的新劣化」，并据此产出五条路线评估表 + 一轮 Owner 授权的错方向。

**真相三层**：
1. **信号本身 07-13 05:54 才存在**——边车 `classifyProviderStop()` 由 `8ca6aa066`(07-13 05:38) 新建、`33159e2f3`(05:54) 接入调用路径。整月 0 是**我方没这段代码**，不是现象不存在。
2. **07-13 前的「全绿」是坏探针伪影**——aliyun 页面 markup 已漂移，runner `page.fill Timeout waiting for input[placeholder*="example"]`（目标域根本没提交成功）却仍报 `done nodes=151`，且**每域恒 151 节点 = 未绑定行**。恒定值是伪影的指纹。
3. **真起点 07-09/10，比表面早 5 天**——两次「暴涨」都只是可见性拐点。

**铁律**：任何「从未发生过的事情突然开始发生」，**先查该信号的采集机制是什么时候建的**（`git log --follow` 判据函数 + 逐 commit 取内容比对判据是否存在），再谈现象。同型陷阱：判据/分类逻辑改动也会造成同样的伪劣化。

## 翻案唯一杠杆 = 跨改动的同口径基线

admin 侧 07-13 前**无可比失败码**（旧码把失败吞成假成功）→ 纯用 admin 日志永远只能在「07-14 还是 07-17」之间打转，而两个都是错的。破局靠找到**边车层 `[probe] error`** 这个横跨代码改动的低层信号：`Jul 05-08=0 → Jul 09=181(~16%) → Jul 10-13=587/699/568/625(~50%)`，一次量出真起点。

**方法**：上层信号因改造而不可比时，往**更低、更少被改动的层**找计数器（边车/nginx/内核/外部服务日志）。找不到同口径基线就别下「变好/变坏」结论。

## 附带教训

- **报比率必须同时报分母与绝对数**：本例「178→355 翻倍」是 HKT 日切 + 二号源融合吸收造成的口径伪影，denied 绝对数只涨 37%（256→350）。先验分母效应第三次咬人，见 [[feedback_distribution_reflects_sample_not_phenomenon]]。
- **上会话报的「178/176」其实是 no_trust 数不是 denied 数**——口径从一开始就串了，后续全部推理建在错数上。引用他人会话的数字必回原文核字段名。
- **探针链路无一层能发现自己瞎了**：主探针假健康报了一个月、二号源 fusion 07-10~14 静默断供，两例都不是靠告警发现的。与 [[reference_n9e_rule_live_redtest]]「凡稳态恒 0 的告警都该做一次 live 红验证」同款，反向补充：**凡「稳态恒满/恒成功」的探测也该定期验它是不是在空转**。
- **三点 diff 的 `-` 侧是合并基不是对比分支**：本轮 lead 据此差点误报「生产与 master 分叉、master 非超集」，直接查两侧真值才纠正。见 [[feedback_verify_not_recall]]。
