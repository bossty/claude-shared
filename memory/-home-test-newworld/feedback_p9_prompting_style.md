## 触发 P9 模式的判断

**适合 P9 的任务**：
- 3+ 个子任务可并行
- 跨模块改动（common + web + admin + frontend）
- 有战略/方案设计成分
- 多日周期、需要灰度/回滚

**不适合 P9**：
- 改一个按钮、修一行 bug → 直接说需求
- 单纯调研 / 问答 → 不要 P9，避免框架累赘
- 一次性脚本 → 按步骤对话即可

## 开场模板（30 秒搞定）

给 Claude 这 6 项能显著提升表现：

```
[业务场景]：500 万 DAU 视频站、CF Tunnel 回源、GFW 下运营（具体业务 + 规模）
[本次目标]：一句话说清要解决什么问题
[已知现象]：从管理后台/日志看到的具体数据（数字 > 描述）
[我的假设]：我怀疑 A/B/C（让 Claude 知道你的判断）
[边界]：现有哪些可以推翻，哪些是红线必须保
[交付期望]：方案文档？Task Prompt？代码？报告？
```

**关键**：具体数字 + 具体痛点 = 可锚定。不是"帮我优化"，而是"池只有 10 个、T+1 似乎没实施、A vs B 怎么选"。

## 推进中的对话方式

### 该决策时给明确答案
- "D3 你决定" 可以（信任）
- "A" / "不，换 B 因为 XXX" 更好（方向 + 理由）
- 避免把决策踢回 Claude："你觉得呢"

### 纠错 / 打断 / 换视角
看到 Claude 在幻觉或跑偏，一句话拉回：
- **"有证据吗"** → 强制跑命令验证
- **"这一步先不做 / 不在本次范围"** → 防过度工程
- **"XXX 不是 N 个吗" / "为什么不是 Y"** → 逼换视角（经典 P9 揪头发）

### 量化验收标准
不说 "可上线"，说 "5xx<0.5% + PV 不跌 5% 就上线" —— Claude 能自主判断。

### 不要过度客气
- ✅ "派 P8 调研 Worker 部署"
- ❌ "你觉得要不要顺便看看 Worker"（省 5 轮对话）

## Claude 常见掉链子模式

老实承认的失效场景：
1. **上下文断了后**（compaction）会把已对齐的决策当新问题再问
2. **没证据时会凭 training data 瞎猜**（需要"事实驱动"类规则压住）
3. **任务不清晰时往"感觉完整"方向堆砌**（不是用户要的方向）

## 用户的工具箱

让 Claude 稳定在 P9 状态的四个抓手：
1. **具体事实**（数字 > 描述，案例 > 抽象）
2. **快速纠错**（发现错误立刻打断）
3. **明确边界**（什么能推翻、什么是红线）
4. **阶段性拍板**（该决策时不犹豫）

## 一次成功案例的特征（2026-04-12 GFW/域名 V2 方案）

具备条件：
- 用户开场给了 3 个具体痛点 + 具体数字（10 个留存域、T+1 未实施、一渠道一域名选型）
- 用户会揪头发（"relay 域名不是多个么？" → 补上 P9 也漏掉的盲区）
- 该决策时给明确答案（D1-D5 + 灰度节奏 + 短链池弹性）
- 该放手时放手（"你来决定" / "我只做 review" / "接管灰度观察"）
- 任务真 P9 尺度（7-10 天、6 个 TP、多 agent 并行、真实生产）

结果：7 天规划、2 个 sprint、10+ 个 P8/P7 并行交付、通宵值守自动 checkpoint、零回滚上线。

## 何时提示 Claude 切回默认

- Claude 在非 P9 尺度任务上套 P9 框架 → "这个不用 P9，直接做"
- Claude 过度仪式感（每轮都有大段旁白）→ "简短汇报就行"
- Claude 反复问同一类问题 → "我之前说过 X，别再问"
