FACT-DRIVEN 红线：**指标名字能误导人，下结论前追源码**。

**Why**：2026-04-17 案例复盘 — 
- 播放诊断仪表盘显示"海外播放失败飙升"
- 我看 `monitor:cdn-failover` Redis 指标 = 0，直接下结论"CDN 没问题，播放健康"
- 实际上 `cdn-failover` 这个 key 前端代码里**根本没有任何地方上报**，所以永远是 0
- 真实播放失败走另一条链路：`/api/v1/edu/playback-error` → MySQL `playback_error_log`
- 导致误导用户，浪费一轮调查时间

**How to apply**：
1. 分析任何 Redis 指标/数据库数据/仪表盘数字之前，先 `grep` 搜代码确认**写入方**
2. 确认"这个 key 是由谁、什么时机、什么条件写入的"
3. 只有当写入逻辑和你的解读假设对齐，才能用这个指标下结论
4. 如果 grep 发现**只有定义没有写入**（或者写入条件和业务事件不挂钩），这个指标就是假信号，必须排除

更广义教训：**读 前 先 验 证 元 数 据**。系统随时间演化，旧指标名字可能被废弃但保留，新功能可能用新 key，名字相同含义不同。
