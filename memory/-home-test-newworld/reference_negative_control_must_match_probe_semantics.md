**2026-07-20/21 GFW P0 ④ 实事故：一条裸 `403` 判据吞掉自家 40% 的成功探测，被误诊为「外部探测源风控劣化」长达 9 天，且一整轮多 agent 取证不但没查出来，还写出了「403 自伤假说判定不成立」的四重闭合论证。**

## 事故本体

边车 `aliyun-probe-runner` 的 `denied` 判据含**无词边界的裸 `403`**，跑在拨测结果**已渲染之后**（`phase=post_render`）且**无成功证据兜底**。阿里云结果表的耗时列 `403ms`、速率列 `403.40KB/s` 命中它 → **整域已拿到的成功探测被当成「服务商拒绝服务」丢弃**（被丢弃行的节点 `HTTPResponseCode` 全是 `200`）。

对照实验（同边车／同批域／同源，唯一变量＝加成功证据兜底）：

| | 修复前 | 修复后 |
|---|---|---|
| `[probe] done` | 79 / 138（57.2%） | 280 / 288（**97.2%**） |
| `provider_stop:denied` | 56（**40.6%**） | **0** |

## 核心教训：负对照口径必须与判据逐字一致

翻案前那条「四重闭合」里最致命的是**②负对照**：声称「正常结果表 152 行 `403` 出现 **0 次**」。而真实结果表里**满是 `403`**。原因是采样口径不一致——**判据是裸子串扫全文，负对照大概率只查了 `HTTPResponseCode` 状态码列**。于是量出一个根本不存在的零，并成为其余三条的地基：

- ① 「per-domain denied 分布拟合 i.i.d. p≈0.4、无 8/8 全拒域」→ **与自伤完全相容**，证伪的是「按域封禁」而非自伤（某轮是否自伤取决于该轮页面是否恰好渲染出含 `403` 的数值，本就近似独立随机）。
- ③ 「目标域名不含 `403`」属实但**无关**（撞的是耗时/速率数值）。
- ④ 「命中 denied ⇒ 页面非正常结果表」是**循环论证**，前提正是②的假零。

**铁律**：证伪一个判据时，负对照必须用**该判据本身**去扫**该判据实际看到的那段文本**。用结构化字段（某一列、某个 key）去否证一个扫全文的正则，只会量出假零。属 [[reference_absence_claims_need_two_independent_probes]] 的特例，也是「零结果先怀疑判据」的又一实例。

## 附带订正

- **`adjacent` 标志区分不了自伤与否**：`403ms` → `adjacent=true`（紧贴字母）、`403.40KB/s` → `adjacent=false`（前空格后点号），**两者都是自伤**。据它估过「82.1% 自伤」，实际 100%。设计观测字段时想清楚它到底可判别什么。
- **`journalctl --utc --since '<裸时间>'`**：`--utc` 只改输出显示，**不改变对裸时间字符串的解析**，直接查会返回 0 条假空；必须写显式 `UTC` 后缀。
- **裸子串误命中一天内出现四次**（`403` 撞 `1403ms`／等待器撞日志正文 `error: redis down`／grep `PROBE_DIAG_SAMPLE=1` 撞禁用警告文本本身／本事故），连独立蓝军自己也栽了一次。见 [[reference_bare_substring_gate_needs_success_evidence_backstop]]。

## 最该警惕的那一条

**「外部环境劣化」是所有解释里最能让人停止排查自家代码的一个。** 它自洽、可归因、还免责，于是取证方向整个转向「量化外部源有多坏」，没人回头问「这 40% 到底是谁判的」。**凡结论落在「外部变坏且我方无改动」，都要专门再问一遍：这个信号是我方哪段代码产出的？那段代码自己可信吗？**

同族：[[reference_signal_onset_is_collector_birth_not_phenomenon]]（把采集器诞生误当现象起点）、[[feedback_distribution_reflects_sample_not_phenomenon]]（分布反映样本构成而非现象特征）——三者都是「观测工具本身在骗你」。
