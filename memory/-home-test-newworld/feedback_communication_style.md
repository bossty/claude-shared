用户 2026-04-23 晚上明确要求并"记入铁律"：**和他沟通时，除非必要情况，不要使用简称，能用中文的词汇尽量用中文**。

**为什么**：用户偏好清晰的中文表达，简称和大量英文夹杂会降低对话可读性。

**如何应用**：

1. 技术术语中**保留英文**的"必要情况"（不算违规）：
   - 工具命令（mvn verify / git commit / curl 等）
   - HTTP 状态码 / headers（403 / Access-Control-Allow-Origin / cf-cache-status 等）
   - 第三方系统名（Cloudflare / NameSilo / Spring / Redis / Mockito 等）
   - 代码符号（类名、方法名、commit SHA、文件路径）
   - 业务约定好的短代号（P 域名 / A 域名 / 事故链 Layer 3 等，因为项目内约定已是短码）

2. **必须翻成中文**的类型：
   - 岗位代号：P9 → 9 级工程师 / 技术负责人；P7 → 7 级工程师；P8 → 8 级工程师；agent → 代理（但"派 agent"保留已流行的说法，或说"派代理"）
   - 项目管理词：scope → 范围；backlog → 待办池；sprint → 冲刺；retrospective → 复盘；TD → 技术债务
   - 研发流程词：HALT → 主动停下 / 停下汇报；runbook → 操作手册；pre-flight → 部署前检查
   - 状态评价：P0/P1/P2 → 优先级 0/1/2（在首次出现时用中文解释一次，后续可保留简写）

3. **段落内高频出现的全名可在首次声明后**简称一次（如首次"Cloudflare 响应头变换规则（以下简称 Transform Rule）"），这算合理使用简写，不是违反规矩。

4. **报告/文档写入 src/main/recon 目录的文字材料不受此规矩限制**，因为这是项目资产而不是当下对话回复。

违反后果：用户再次提醒即视为规矩执行失败，需自查并改写整段回复。
