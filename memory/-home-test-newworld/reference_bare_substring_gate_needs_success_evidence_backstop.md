**适用面远超 GFW**：任何「对页面/响应/日志做文本匹配，命中即判失败并丢弃整个结果」的判据（风控识别、错误分类、健康检查、爬虫反爬检测）。

## 核心

**裸子串判据的误命中不是概率问题，是结构问题**——它取决于**在哪个调用点跑**：

- **有成功证据兜底的调用点**（例：只在「零结果」时才跑判据）→ 即使判据写宽也吃不掉一次成功结果，最坏是把真失败归错类。
- **无成功证据兜底的调用点**（例：提交前跑、每轮都跑）→ 判据表里**每一条裸子串都是全额风险**，命中即误杀一次本可成功的探测。

**铁律：往这类判据表加词，必须先问「它会在哪些调用点跑、那里有没有成功证据兜底」。** 加词的人通常只想着「这个词是不是风控文案」，不会想到调用点差异。

## 2026-07-20 三个实例（同源，只是换了源）

1. **aliyun `denied` 判据 `/403|forbidden|.../i` 无词边界**：结果表把每节点 HTTP 状态码渲染进正文，`1403ms` 这类耗时数值与真 `403 Forbidden` 命中完全一样。（该假说最终经四重证据判定**非主导机制**，但机制本身成立，代码层面仍在。）
2. **tcptest 频控判据漏裸文案**：原写 `访问(?:过于|太)频繁` 强制要求程度词，野外最常见的「访问频繁，请稍后再试」直接漏 → 风控被当成源站死亡。**红测抓出**。
3. **tcptest 判据跑在「关闭所有广告」之前**：该页有广告位（代码自己要点关闭按钮），广告文案「您的网络访问受限？装我们的加速器」命中裸子串 `访问受限` → **整域被判 denied 丢弃**。修法=判据后移到关广告之后 + 反向守卫测试锁「关广告后风控仍在则照常 hard-stop 且不 fill」（防后移过头变成形同虚设）。

注意 2 和 3 的对称性：**同一个人给「频繁」加了紧邻词约束，却没意识到「访问受限」「已被限制」是同一类裸子串、缺同样保护**。局部修补挡不住这类 bug，要在调用点层面设防。

## ★ 结构性防护优于把正则写准

最有效的一招不是把判据写得多准，而是**限制判据能在什么条件下开火**：

> tcptest `post_wait` 判据**仅在 `nodes.length === 0` 时才跑**——拿到节点即成功证据，此时页面必然渲染了各节点状态码，再跑文本判据只会误杀。**即使判据将来被写宽，也吃不掉一次成功探测。**

同理适用：只在超时/空响应/非 2xx 时才跑错误文本分类，而不是无条件对每个响应体做匹配。

## 附带：判据没有真样本 = 结论不可证伪

`TCPTEST_STOP_RULES` 文案全部照一份 POC 报告描述推得，**无一条来自真实风控页**。失败模式静默：措辞不匹配 → 落回 `no_results` → **没有任何信号提示判据没生效**。而下游 canary 判据恰是「零 captcha」→ **判据打空会让 canary 误报通过**（零 captcha 不是没被风控，是没认出来），比误报失败更坏。

**修法**（已实现）：加默认关的诊断取样开关，命中与未命中都落**脱敏有界**样本 + `classified=<stop|none>` 字段；`classified=none` 即判据盲区样本，直接就是要补进判据表的词。**首轮验证跑当取样轮而非判定轮。**

取样的隐私边界：取样行**刻意不带 domain 字段**（该行会 dump 页面文本，不带域名则即使日志外泄也只含对方措辞、不含我方域名清单），关联靠同轮相邻日志行。
