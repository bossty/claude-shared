看到「故障/失败高度集中在某维度（某国家/某源站/某模块/某时段）」，**在归因前必须先问：这个样本源里，该维度的基线占比本来是多少？** 单一样本源的分布首先反映的是**样本构成**，只有与基线之比才是现象特征。

**Why**：2026-07-16 BL-72 实事故——「504 的 94% 是 CN 客户端」被连续两个会话读作「GFW 针对性干扰 CN 跨境连接」，据此写进 RCA、memory 判别法、BACKLOG 收官结论。真相：证据全取自 canary 单 zone（`eduspace181.link`），而该 zone 的**真实用户本来就 94% 在 CN**——分子分母同源，那个 94% 什么也没证明。换成流量全球分布的主域（17.rip）跑一次 `country×status`：**US 504=38.9% > CN 23.9%**（DE 29%/CA 36.6%/JP 43.4%），GFW 假说当场证伪——GFW 干扰不到美国。代价：两轮排查 + 一次错误收官 + 一条误导性 memory 判别法（该 memory 的「CN 集中先想 GFW」引导，正是把第二轮引进沟里的东西）。

**How to apply**：
1. 下任何「集中在 X」的归因前，先取一个**构成不同的对照样本源**跑同一查询（本项目：canary zone ↔ 主域 17.rip；单爬虫源 ↔ 全源；单节点 ↔ 全舰队）。通常一次查询、不到一分钟。
2. 报比率不报绝对数：「CN 占失败的 94%」无意义，「CN 失败率 23.9% vs US 38.9%」才有意义。分母必须是同群体的总量。
3. 反向自检问句：**「这个维度上，成功的那批是什么分布？」**——若失败分布 ≈ 成功分布，那就只是流量构成，不是故障特征（本例中 SG 504 率 0% vs US 38.9% 才是真信号）。
4. 同类高发场景：爬虫失败按源站、错误日志按模块、投诉按地区、慢查询按接口——凡「按 X 分组看故障」皆适用。
